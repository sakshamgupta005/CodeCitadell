Design Rationale

The signature element of FixPilot is the Diagnostic Page's living Investigation Thread — a vertical timeline on the left rail that builds in real-time as the AI reasons through possible causes. It makes the thinking process visible, not just the conclusion. This is what separates FixPilot from any chatbot wrapper. Every other design choice is quiet and disciplined to let this element be the memorable thing.

Why dark-first?

Technical audiences (engineers, field technicians) associate dark UI with professionalism and precision — the same reason Figma, GitHub, Linear, and VS Code default dark. It also makes the indigo/teal accent colors pop at maximum contrast.

Why Fraunces only for heroes?

A display serif in one place creates contrast and character without feeling like a design accident. Using it everywhere would dilute it. The tension between editorial display type and tight mono data type signals "premium tech tool."

Why teal for diagnostic states?

Teal is semantically distinct from indigo (navigation) and amber (warnings). It reads as "active investigation" — clinical, precise, alive. The pulsing teal indicator on the investigation thread gives the page biological energy.

Why mono font for data?

JetBrains Mono for percentages, citations, and session IDs signals that this data is precise and machine-generated — not marketing copy. It builds trust in the AI's outputs by visually encoding "this is technical truth."

04 Core Feature · Most Important

Diagnostic Assistant

This is the heart of FixPilot. Three columns: the Investigation Thread (AI reasoning made visible), the Chat (conversation), and the Analysis Panel (live evidence). Together they make a judge feel like they're watching a real technician work.

The Investigation Thread is the signature element. It's a vertical timeline on the left that grows and updates in real-time as the AI narrows its analysis. Each step in the diagnostic process — symptom intake, cause generation, question phase, inspection, elimination, root cause — appears as a node on the thread, with a pulsing teal dot showing the current stage. This makes AI reasoning visible, not just a loading spinner.

Left Column: Investigation Thread

  • Fixed 260px width sidebar
  • Vertical timeline with dot-connector pattern
  • 3 dot states: done (indigo filled), active (teal pulsing), pending (empty border)
  • 3 connector states: done (indigo line), active (teal), pending (border color)
  • Steps: Symptom Intake → Cause Generation → Question Phase → Inspection → Elimination → Root Cause → Resolution
  • Each step shows its summary text when complete
  • Active step label in teal, done in indigo

Center Column: Chat Interface

  • Flex column: header → messages → input
  • AI avatar: gradient circle (indigo→teal), shows 🧠
  • AI bubbles: bg-surface background, left-rounded corner square
  • User bubbles: indigo bg, right-rounded corner square
  • Citation chips below AI messages in mono font
  • Follow-up question: amber left-border callout below bubble
  • Quick-reply chips above input for common answers
  • Input bar: attachment icon + send button

Right Column: Analysis Panel

  • 220px wide fixed sidebar
  • Three sections: Suspected Causes / Inspections / Citations
  • Cause bars: horizontal progress bars color-coded by confidence (amber=high, indigo=medium, gray=low)
  • Eliminated causes shown struck-through with red ✕
  • Inspection steps numbered in indigo circles
  • Source citations are tappable links to doc viewer

Visual Communication Strategy

  • Investigation: Teal pulsing dot = active thinking
  • Diagnosis: Confidence bars update after each answer
  • Root cause: Final thread node glows indigo on completion
  • Expertise: Mono font on all data signals precision
  • Elimination: Struck-through cause with red dot = deduction
  • Sources: Every claim has a clickable citation chip
Hackathon Demo Tip: For the demo, pre-load the diagnostic session with the first two steps already complete so judges immediately see the Investigation Thread in an active state. The visual impression of a system already thinking is worth 30 seconds of explanation.

08 System

Shared Components & Interactions

These components appear across multiple pages. Define them once in shadcn/ui with token-based variants.

Button Variants

  • Primary: bg-indigo, white text, hover: brightness(1.1)
  • Secondary: bg-elevated, border, text-primary
  • Ghost: transparent, text-secondary, hover: bg-elevated
  • Danger: bg-red/10, red text, border-red/20
  • Success: bg-green/10, green text
  • All: 8px border-radius, 500 weight, 12–13px font-size

Status Badge Variants

  • Live: green bg/10, green text, green border/20
  • Indexing: amber bg/10, amber text — with pulse dot
  • Processing: indigo bg/10, indigo text
  • Error: red bg/10, red text
  • Draft: bg-elevated, text-muted
  • All: 10px font, 700 weight, 3px border-radius, uppercase

Input Fields

  • Background: bg-elevated
  • Border: border token, focus → indigo
  • Placeholder: text-muted
  • Text: text-primary
  • Error: red border + error message below
  • Font-size: 14px, line-height: 1.5

Toast Notifications

  • Position: bottom-right, z-50
  • Success: left border green, green icon
  • Error: left border red, red icon
  • Info: left border indigo, indigo icon
  • Auto-dismiss: 4s, slide-out animation
  • Background: bg-surface, border

Interaction Principles

Hover states: All interactive elements have a defined hover: either background shift (bg-surface → bg-elevated), border color change (→ indigo at 40% opacity), or transform (cards: translateY(-2px)). Never rely on cursor change alone.
Loading states: Never show a blank screen. Use skeleton loaders that match the shape of the content. The Diagnostic page chat uses a typing indicator (3-dot pulse in teal) while AI responds.
Motion: Use CSS transitions at 150–200ms ease for micro-interactions. The Investigation Thread nodes use a 300ms fade-in as they appear. Avoid animating things the user didn't trigger — the teal pulse on the active node is the only ambient animation.
Responsive breakpoints: Desktop-first but fully mobile-optimized. At ≤768px: sidebar nav collapses to bottom tab bar; 3-column diagnostic collapses to stacked tabs (Chat primary, Investigation and Analysis as swipeable panels); product grid collapses to 1-column.
